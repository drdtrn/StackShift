Before starting any task in this project, you MUST:

1. Read `src/frontend/CURRENTSTATE.md` if the task touches the frontend
2. Read `src/backend/CURRENTSTATE.md` if the task touches the backend
3. Read both if the task is fullstack or unclear

These files contain the authoritative record of what has been built, what is pending, the API contract both sides must agree on, key constraints, and known gotchas. Do not rely solely on code inspection — the state files capture decisions and context that the code alone does not.

After completing any task, you MUST update the relevant state file(s):

- Mark completed cards from 🔲 to ✅ with a brief note
- Add any new files of note to the Key Files table
- Add any new constraints, gotchas, or breaking changes discovered
- Add any new environment variables introduced
- Update the Pending Work section — remove completed items, add newly discovered ones
- Update the "Last updated" date at the top

Keep updates factual and terse — one line per item is almost always enough. The state files are a living document, not a log.
